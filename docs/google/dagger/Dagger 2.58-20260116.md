Potential breaking changes:

* Flipped default for <code>dagger.useBindingGraphFix</code> to <code>enabled</code> (<a class="commit-link" data-hovercard-type="commit" data-hovercard-url="https://github.com/google/dagger/commit/d41e8362b963d3340175824e8844c770b781b133/hovercard" href="https://github.com/google/dagger/commit/d41e8362b963d3340175824e8844c770b781b133"><tt>d41e836</tt></a>).  
  See <a href="https://dagger.dev/dev-guide/compiler-options#useBindingGraphFix" rel="nofollow">https://dagger.dev/dev-guide/compiler-options#useBindingGraphFix</a>.  
  <strong>Fix:</strong> If this change breaks you it’s likely due to providing a binding in an incorrect module, (e.g. the binding requests an Activity but is installed in the SingletonComponent). The fix is to move the <code>@Provides</code> so that it is installed into the correct component (e.g. the ActivityComponent instead of SingletonComponent). You can also set <code>dagger.useBindingGraphFix=disabled</code> but note that this flag is only temporary and will eventually be removed.
* The use of <code>abstract var</code> in components is now banned in super types.  
  The reason for this change is that an <code>abstract var</code> property creates both a getter and a setter on the component which is almost always unintentional.  
  <strong>Fix:</strong> If this change breaks you, use either an <code>abstract val foo: Foo</code> if you need a getter or <code>abstract fun inject(foo: Foo)</code> if you need an inject method. Note that <code>abstract var</code> is already banned if it is declared directly on the <code>@Component</code> class/interface, but this change fixes the check so that it’s banned even if it’s inherited from a super type.
