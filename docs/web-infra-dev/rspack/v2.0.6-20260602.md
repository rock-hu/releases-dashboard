Highlights

React Server Components: enforce <code>server-only</code> / <code>client-only</code> boundaries

RSC builds now validate the <code>server-only</code> and <code>client-only</code> marker packages. Importing a <code>client-only</code> module from a server-graph module (or a <code>server-only</code> module from a client-graph module) is reported as a diagnostic. This marker-package validation always runs and is independent of <code>disableClientApiChecks</code>, which now only scopes the React client-API checks.

    // server component (server graph)
    import 'client-only'; // ❌ reported: client-only imported in a server module

    // client component (client graph)
    import 'server-only'; // ❌ reported: server-only imported in a client module

Expose <code>DeterministicModuleIdsPlugin</code>

The webpack-compatible deterministic module id plugin is now exposed through the JS API as <code>rspack.ids.DeterministicModuleIdsPlugin</code>, giving you fine-grained control over module ids for long-term caching. It supports the webpack options <code>test</code>, <code>maxLength</code>, <code>salt</code>, <code>fixedLength</code>, and <code>failOnConflict</code>.

    const rspack = require('@rspack/core');

    module.exports = {
      optimization: { moduleIds: false },
      plugins: [
        new rspack.ids.DeterministicModuleIdsPlugin({
          maxLength: 5,
          failOnConflict: true,
        }),
      ],
    };

CSS Modules: preserve <code>composes</code> cascade order

CSS Modules now use a shared topological sort to keep the source order of <code>composes</code>, so composed declarations land in the output cascade in the order you authored them. This fixes subtle specificity surprises when composing local class names across modules.
