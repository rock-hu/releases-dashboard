<em>Sourced from <a href="https://github.com/redis/go-redis/releases">github.com/redis/go-redis/v9's releases</a>.</em>
>
> 9.21.0
>
> This is a minor release adding new features and bug fixes. There are
> no breaking changes; upgrading from 9.20.x is a drop-in replacement.
>
> 🚀 Highlights
>
> Zero-copy <code>GetToBuffer</code> / <code>SetFromBuffer</code>
>
> Two new <code>StringCmdable</code> methods let callers read and write
> Redis string values directly into and from pre-allocated byte buffers,
> eliminating the per-call payload allocation that
> <code>Get</code>/<code>Set</code> incur:
>
>     GetToBuffer(ctx, key, buf) *ZeroCopyStringCmd //
>     reads into buf; ZeroCopyStringCmd { Val() int; Bytes() []byte; Result()
>     (int, error) }
>     SetFromBuffer(ctx, key, buf) *StatusCmd
>
> <code>GetToBuffer</code> decodes the bulk reply straight into the
> caller-owned <code>buf</code> (no intermediate allocation); a buffer
> that is too small returns an error after draining the payload, so the
> connection stays aligned for the next reply. <code>SetFromBuffer</code>
> is provided for API symmetry — it dispatches to the same
> <code>[]byte</code> writer path as <code>Set(ctx, key, buf, 0)</code>
> and produces byte-identical output on the wire. Available on
> <code>*Client</code>, <code>*ClusterClient</code>, <code>*Ring</code>,
> <code>*Conn</code> and <code>Pipeliner</code>.
>
> (<a href="https://redirect.github.com/redis/go-redis/pull/3834">#3834</a>)
> by <a href="https://github.com/ndyakov"><code>@​ndyakov</code></a>
>
> Explicit <code>LIMIT 0</code> for stream trimming
>
> Redis treats <code>XTRIM</code>/<code>XADD</code> approximate-trim
> (<code>~</code>) <code>LIMIT 0</code> as "disable the trimming
> effort cap entirely", which differs from omitting
> <code>LIMIT</code> (the implicit <code>100 * stream-node-max-entries</code> default). The command builders previously
> only emitted <code>LIMIT</code> when <code>limit &gt; 0</code>, so
> callers could never send an explicit <code>LIMIT 0</code>. Following the
> <code>KeepTTL = -1</code> precedent, the new <code>XTrimLimitDisabled = -1</code> sentinel now emits an explicit <code>LIMIT 0</code>;
> <code>limit == 0</code> keeps the historical no-<code>LIMIT</code>
> behavior, so existing callers produce byte-identical commands.
>
> (<a href="https://redirect.github.com/redis/go-redis/pull/3848">#3848</a>)
> by <a href="https://github.com/TheRealMal"><code>@​TheRealMal</code></a>
>
> ✨ New Features
>
> * <strong>Zero-copy buffer string commands</strong>: new <code>GetToBuffer</code> / <code>SetFromBuffer</code> on <code>StringCmdable</code> and the <code>ZeroCopyStringCmd</code> result type, reading/writing string values into caller-owned buffers without per-call payload allocation (<a href="https://redirect.github.com/redis/go-redis/pull/3834">#3834</a>) by <a href="https://github.com/ndyakov"><code>@​ndyakov</code></a>
> * <strong><code>XTrimLimitDisabled</code> sentinel</strong>: <code>XTRIM</code>/<code>XADD</code> approximate trimming can now send an explicit <code>LIMIT 0</code> to disable the trim effort cap, via the new <code>XTrimLimitDisabled = -1</code> sentinel (<a href="https://redirect.github.com/redis/go-redis/pull/3848">#3848</a>) by <a href="https://github.com/TheRealMal"><code>@​TheRealMal</code></a>
> * <strong>PubSub health-check timeouts</strong>: <code>channel.initHealthCheck</code> now bounds the <code>Ping</code> it issues with a fresh per-check timeout context (the exported <code>pingTimeout</code> / <code>reconnectTimeout</code>) instead of <code>context.TODO()</code>, so a stuck health-check Ping can no longer block indefinitely (<a href="https://redirect.github.com/redis/go-redis/pull/3819">#3819</a>) by <a href="https://github.com/abdellani"><code>@​abdellani</code></a>
> * <strong>Skip redundant <code>UNWATCH</code> in <code>Tx.Close</code></strong>: a transaction now tracks whether a <code>WATCH</code> is still active (<code>watchArmed</code>) and only issues <code>UNWATCH</code> on <code>Close</code> when it is, removing an extra round trip on the common <code>WATCH</code>/.../<code>EXEC</code> and no-key <code>Watch</code> paths while never returning a connection to the pool with an active watch (<a href="https://redirect.github.com/redis/go-redis/pull/3854">#3854</a>) by <a href="https://github.com/fcostaoliveira"><code>@​fcostaoliveira</code></a>
>
> 🐛 Bug Fixes
>
> * <strong><code>maintnotifications</code> <code>ModeAuto</code> fail-open</strong>: <code>ModeAuto</code> now stays fail-open when the server does not support maintenance notifications — connections are retired and tracking is guarded during downgrade so the client keeps working instead of erroring (<a href="https://redirect.github.com/redis/go-redis/pull/3853">#3853</a>) by <a href="https://github.com/terrorobe"><code>@​terrorobe</code></a>
>
