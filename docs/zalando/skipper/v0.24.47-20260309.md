Context

Operators in multitenant ingress setups currently have no programmatic  

way to discover which routes failed validation — the only option is  

grepping logs for <code>"failed to process route"</code> messages. The <code>routeTable</code>  

already tracks <code>invalidRoutes []*eskip.Route</code> alongside <code>validRoutes</code>,  

but <code>Routing.ServeHTTP</code> unconditionally serves only the valid set.  

Invalid routes are silently dropped from the routing table with no API  

visibility into what broke or why.

Approach

This extends the existing <code>/routes</code> handler with a <code>?invalid=true</code> query  

parameter rather than adding a separate endpoint. The query param  

approach follows <a class="user-mention notranslate" data-hovercard-type="user" data-hovercard-url="/users/szuecs/hovercard" data-octo-click="hovercard-link-click" data-octo-dimensions="link_type:self" href="https://github.com/szuecs">@szuecs</a>'s guidance in the issue — invalid routes are a  

filtered view of the same data, not a distinct resource, and this avoids  

new mux registrations or a parallel <code>ServeHTTP</code> implementation.

The harder design decision was how to expose error messages (per  

<a class="user-mention notranslate" data-hovercard-type="user" data-hovercard-url="/users/AlexanderYastrebov/hovercard" data-octo-click="hovercard-link-click" data-octo-dimensions="link_type:self" href="https://github.com/AlexanderYastrebov">@AlexanderYastrebov</a>'s suggestion). The validation errors are computed in  

<code>processRouteDefs</code> and <code>newMatcher</code> but were previously only sent to  

metrics and discarded. Rather than adding an error field to  

<code>eskip.Route</code> (a core type used across the entire codebase for parsing,  

serialization, and matching), the errors are stored in a separate  

<code>invalidRouteErrors map[string]string</code> on <code>routeTable</code> and merged into  

the JSON response via a custom <code>MarshalJSON</code> on <code>InvalidRouteResponse</code>.

Changes

* <code>processRouteDefs</code> returns a third value — error messages keyed by  
  route ID — collected from filter, predicate, backend, and matcher  
  validation failures
* <code>routeTable</code> gains an <code>invalidRouteErrors</code> field, populated from both  
  <code>processRouteDefs</code> and <code>newMatcher</code> error paths in <code>receiveRouteMatcher</code>
* <code>Routing.ServeHTTP</code> switches between <code>validRoutes</code> and <code>invalidRoutes</code>  
  based on the <code>invalid</code> query param; all existing behavior (pagination,  
  content negotiation, <code>X-Count</code>/<code>X-Timestamp</code> headers) works identically  
  for both sets
* <code>InvalidRouteResponse</code> wraps a route with its error string, using  
  custom JSON marshaling to merge the error into the route's existing JSON  
  structure (necessary because <code>eskip.Route</code> has its own <code>MarshalJSON</code>)
* Documentation added to the "Routing table information" section of  
  <code>operation.md</code>

Testing

Five new tests cover the feature surface: JSON response with error field  

validation, eskip text format output, HEAD request with correct count,  

pagination with offset/limit, and backward compatibility (no param,  

<code>?invalid=false</code>, and <code>?invalid=</code> all return valid routes). All eleven  

<code>TestRoutingHandler*</code> tests pass, including the six pre-existing ones —  

no regressions.

Notes for Reviewer

* The <code>InvalidRouteResponse.MarshalJSON</code> round-trips through  
  <code>json.Marshal</code> → <code>map[string]json.RawMessage</code> → inject error →  
  <code>json.Marshal</code>. This is slightly less efficient than direct byte  
  manipulation but avoids fragile assumptions about <code>eskip.Route</code>'s JSON  
  output format. The cost is negligible since this endpoint is only hit by  
  operators debugging, not in the request path.
* The issue mentions adding "a new metric for number of invalid routes"  
  — individual route metrics already exist via <code>SetInvalidRoute</code> and the  
  gauge tracking added in <a class="issue-link js-issue-link" data-error-text="Failed to load title" data-id="3209530346" data-permission-text="Title is private" data-url="https://github.com/zalando/skipper/issues/3550" data-hovercard-type="pull_request" data-hovercard-url="/zalando/skipper/pull/3550/hovercard" href="https://github.com/zalando/skipper/pull/3550">#3550</a>/<a class="issue-link js-issue-link" data-error-text="Failed to load title" data-id="3372930666" data-permission-text="Title is private" data-url="https://github.com/zalando/skipper/issues/3610" data-hovercard-type="pull_request" data-hovercard-url="/zalando/skipper/pull/3610/hovercard" href="https://github.com/zalando/skipper/pull/3610">#3610</a>, so this PR focuses solely on the  
  API surface.
* Scope excludes routesrv (<code>cmd/routesrv</code>), which has its own route  
  handler and would need separate treatment.
