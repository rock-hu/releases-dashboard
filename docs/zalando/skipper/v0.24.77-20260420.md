> How to prepare a release [!NOTE]  
> >
> > Relevant for maintainers with write access only.
>
> 1. Switch to a new branch from <code>main</code>.
> 2. Run <code>npm test</code> to ensure all tests are passing.
> 3. Update the version in <a href="https://github.com/actions/cache/blob/main/package.json"><code>https://github.com/actions/cache/blob/main/package.json</code></a>.
> 4. Run <code>npm run build</code> to update the compiled files.
> 5. Update this <a href="https://github.com/actions/cache/blob/main/RELEASES.md"><code>https://github.com/actions/cache/blob/main/RELEASES.md</code></a> with the new version and changes in the <code>## Changelog</code> section.
> 6. Run <code>licensed cache</code> to update the license report.
> 7. Run <code>licensed status</code> and resolve any warnings by updating the <a href="https://github.com/actions/cache/blob/main/.licensed.yml"><code>https://github.com/actions/cache/blob/main/.licensed.yml</code></a> file with the exceptions.
> 8. Commit your changes and push your branch upstream.
> 9. Open a pull request against <code>main</code> and get it reviewed and merged.
> 10. Draft a new release <a href="https://github.com/actions/cache/releases">https://github.com/actions/cache/releases</a> use the same version number used in <code>package.json</code>
>     1. Create a new tag with the version number.
>     2. Auto generate release notes and update them to match the changes you made in <code>RELEASES.md</code>.
>     3. Toggle the set as the latest release option.
>     4. Publish the release.
> 11. Navigate to <a href="https://github.com/actions/cache/actions/workflows/release-new-action-version.yml">https://github.com/actions/cache/actions/workflows/release-new-action-version.yml</a>
>     1. There should be a workflow run queued with the same version number.
>     2. Approve the run to publish the new version and update the major tags for this action.
