# v1.46.0

## release on 20251219
## description
## changes
These are the release notes for the v1.46.0 release of <a href="https://backstage.io/" rel="nofollow">Backstage</a>.

A huge thanks to the whole team of maintainers and contributors as well as the amazing Backstage Community for the hard work in getting this release developed and done.

Highlights

To begin with, this release has a few important framework level updates.

<strong>BREAKING:</strong> Updated supported Node.js versions to 22 and 24

This release moves up the supported Node.js versions to 22 and 24, according to our <a href="https://backstage.io/docs/overview/versioning-policy#nodejs-releases" rel="nofollow">versioning policy</a>. As part of this change, we are also setting the compilation targets to ES2023, which lines up with the supported Node.js versions.

The default TypeScript <code>target</code> compiler option is however left at ES2022. This is because older versions of TypeScript will error if they see an unsupported value, even if an override is set in your local <code>tsconfig.json</code>. You can still choose to set the TypeScript <code>target</code> to either <code>ES2023</code> or even <code>ES2024</code> in your own <code>tsconfig.json</code> if you prefer.

<strong>BREAKING:</strong> Switched modules to <code>ES2020</code> and resolution mode to <code>bundler</code>

Now that the ecosystem has had some time to mature and adapt, we have finally switched over the default module resolution mode to <code>bundler</code> instead of <code>node</code>. The <code>module</code> setting was also changed to <code>ES2022</code>.

You <em>may</em> need to bump some dependencies as part of this change, or fix imports in code. The most common source of this is that type checking will now consider the <code>exports</code> field in <code>package.json</code> when resolving imports. This in turn can break older versions of packages that had incompatible <code>exports</code> fields. Generally these issues will have already been fixed in the upstream packages, but if you depend on old versions or unmaintained packages, you may still notice some effects of this.

You might be tempted to use <code>--skipLibCheck</code> to hide issues due to this change, but it will weaken the type safety of your project. If you run into a large number of issues and want to keep the old behavior, you can reset the <code>moduleResolution</code> and <code>module</code> settings of your own <code>tsconfig.json</code> file to <code>node</code> and <code>ESNext</code> respectively. But keep in mind that the <code>node</code> option will be removed in future versions of TypeScript.

A future version of Backstage will make these new settings mandatory, as we move to rely on the <code>exports</code> field for type resolution in packages, rather than the <code>typesVersions</code> field.

<strong>BREAKING:</strong> Moving toward updating <code>jest</code> and its DOM

The <code>jest</code> test framework used to be a builtin dependency of the Backstage CLI. In this release, it has been made into a peer dependency to allow for easy upgrades to newer versions as you see fit.

If you run tests using the Backstage CLI, you must therefore add Jest and its environment dependencies as <code>devDependencies</code> in your project root <code>package.json</code> when you upgrade.

You can choose to install either Jest 29 or Jest 30. The built-in Jest version before this change was Jest 29, however, we recommend that you switch to Jest 30. Upgrading will solve the <code>Could not parse CSS stylesheet</code> errors, allow you to use MSW v2 in web packages, and ensure that you remain compatible with future versions of the Backstage CLI. Support for Jest 29 is temporary, with the purpose of allowing you to upgrade at your own pace, but it will eventually be removed.

* <strong>Jest 29</strong>:

      # in your repository root, run:
      yarn add --dev jest@^29 @types/jest@^29 jest-environment-jsdom@^29

  Pros: No migration needed

  Cons: You may see <code>Could not parse CSS stylesheet</code> warnings/errors when testing components from <code>@backstage/ui</code> or other packages using CSS <code>@layer</code> declarations
* <strong>Jest 30</strong>:

      # in your repository root, run:
      yarn add --dev jest@^30 @types/jest@^30 @jest/environment-jsdom-abstract@^30 jsdom@^27

  Pros: Fixes the stylesheet parsing warnings/errors

  Cons: Requires migration steps in your tests

See the <a href="https://backstage.io/docs/tutorials/jest30-migration" rel="nofollow">Jest 30 migration guide</a> for detailed migration instructions.

<strong>BREAKING:</strong> Now using correct configuration options for Valkey

If you are using Valkey as your backend cache provider, take note. It used to piggyback on the Redis configuration format, but that was not quite correct. So its config options have been updated to better match its capabilities.

So if you use Valkey, you need to upgrade your <code>app-config.yaml</code>:

    backend:
      cache:
        store: valkey
        connection: ...
        client:
    -     namespace: 'my-app'
    -     keyPrefixSeparator: ':'
    +     keyPrefix: 'my-app:'
    -     clearBatchSize: 1000
    -     useUnlink: false

Contributed by <a href="https://github.com/benjidotsh">@benjidotsh</a> in <a href="https://github.com/backstage/backstage/pull/31497" data-hovercard-type="pull_request" data-hovercard-url="/backstage/backstage/pull/31497/hovercard">#31497</a>

<strong>BREAKING:</strong> Techdocs addon test utils removed explicit <code>screen</code>

<code>TechdocsAddonTester.renderWithEffects()</code> used to return a <code>screen</code> that you could use for things like <code>getByText</code>. In order to support moving to newer versions of <code>@testing-library</code>, we have removed this, and you should import and use the normal <code>screen</code> from <code>@testing-library/react</code> instead.

However: that <code>screen</code> canâ€™t see inside the shadow DOM, which TechDocs uses. So if your tests need to look inside shadow DOM, you must install <code>shadow-dom-testing-library</code> and use its <code>screen</code> which provides special shadow-DOM queries, such as <code>getByShadowText</code> instead of `getByText.

Example:

    import { screen } from 'shadow-dom-testing-library';
    // ... render the addon ...
    await TechDocsAddonTester.buildAddonsInTechDocs([<AnAddon />])
      .withDom(<body>TEST_CONTENT</body>)
      .renderWithEffects();
    expect(screen.getByShadowText('TEST_CONTENT')).toBeInTheDocument();

<strong>BREAKING:</strong> Backstage-UI advances!

Again, Backstage-UI has a bunch of updates! One of them is breaking: If you were using the <code>Cell</code> component with text-specific props (<code>title</code> etc), you should now instead use the <code>CellText</code> component for that. Plain <code>Cell</code> is now meant to be used with generic child elements instead.

Simplified route compatibility

If you are partway through a migration to the New Frontend System, you may enjoy the fact that the <code>useApp</code> and <code>useRouteRef</code> hooks now are forwards compatible with NFS. Along with the previous route reference changes this means that there is no longer a need to use <code>compatWrapper</code> from <code>@backstage/core-compat-api</code> to make code based on <code>@backstage/core-plugin-api</code> compatible with <code>@backstage/frontend-plugin-api</code> APIs.

Support for Bitbucket Cloud OAuth

Support for Bitbucket Cloud OAuth has been added - which was much welcome. This introduces an alternative authentication method using a workspace OAuth consumer, alongside App Passwords (deprecated) and API tokens. OAuth does not require a bot or service account and avoids token expiry issues.

As part of this change, some breaking changes were made to low level auth getter functions, such that they now return a promise instead of a plain value.

Contributed by <a href="https://github.com/jksmth">@jksmth</a> in <a href="https://github.com/backstage/backstage/pull/31848" data-hovercard-type="pull_request" data-hovercard-url="/backstage/backstage/pull/31848/hovercard">#31848</a>

New Catalog actions

There are two new MCP-usable actions on the block for registering and unregistering entities: <code>catalog:register-entity</code> and <code>catalog:unregister-entity</code>.

Contributed by <a href="https://github.com/gabemontero">@gabemontero</a> in <a href="https://github.com/backstage/backstage/pull/32042" data-hovercard-type="pull_request" data-hovercard-url="/backstage/backstage/pull/32042/hovercard">#32042</a>

New <code>RepoOwnerPicker</code> for Scaffolder

A new field extension for selecting an organization or owner for a particular repository has been shipped! Currently, this only supports GitHub autocomplete, but contributions welcome in making this available for other providers!

Contributed by <a href="https://github.com/benjidotsh">@benjidotsh</a> in <a href="https://github.com/backstage/backstage/pull/32105" data-hovercard-type="pull_request" data-hovercard-url="/backstage/backstage/pull/32105/hovercard">#32105</a>

Security Fixes

This release does not contain any security fixes.
